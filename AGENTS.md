# AGENTS.md

오박사만능인테리어 정적 홈페이지 운영 및 유지보수 기준서

버전: v3.1

적용 대상: `obaksa-home` 프로젝트

최종 목적: 기존 홈페이지를 깨뜨리지 않고, 시공사례와 콘텐츠를 안정적으로 누적하며, 모바일 방문자가 편하게 문의할 수 있는 구조를 유지한다.

---

## 01 프로젝트 운영 철학

- 최우선 목표는 새 기능 추가가 아니라 기존 홈페이지를 절대 깨뜨리지 않는 것이다.
- 안정성이 기능보다 우선한다.
- 수정 범위는 최소화한다.
- 기존 동작은 반드시 유지한다.
- 사용자가 요청하지 않은 구조 변경, 디자인 리뉴얼, 대규모 리팩토링은 금지한다.
- 작은 수정도 실제 고객 문의와 연결되는 사업 자산으로 본다.

---

## 02 AGENTS 우선순위

- `AGENTS.md`는 이 프로젝트의 최상위 기준 문서다.
- 다른 문서와 충돌하면 `AGENTS.md`를 우선 적용한다.
- `SITE_UPDATE_GUIDE.md`는 참고 문서로만 사용한다.
- 새 파일이 추가되거나 프로젝트 구조, 빌드 프로세스, 배포 방식이 변경되면 먼저 `AGENTS.md`를 갱신한다.
- `AGENTS.md`와 실제 프로젝트 구조가 다르면 작업 전에 차이점을 보고한다.
- 이 문서를 읽지 않았거나 실제 구조와의 정합성을 검증하지 않았다면 작업을 시작하지 않는다.

## 03 작업 시작 절차

1. `AGENTS.md` 전체 읽기
2. `git status --short` 실행
3. 프로젝트 루트 파일 및 폴더 구조 확인
4. 주요 파일 목록과 실제 파일 목록 비교
5. 누락 파일, 신규 파일, 비정상 구조 변경 발견 시 선보고
6. 다중 파일 수정, 공통 CSS/JS 수정, 배포/동기화 작업이 포함될 경우 상위 폴더에 원본 전체 복사 백업 생성
7. 작업 착수 전 단문 보고

```text
[AGENTS 확인 완료]

프로젝트: 오박사만능인테리어 정적 홈
파일 수: [확인된 파일 개수]
추가 파일: [신규 추가된 파일 목록 또는 없음]
누락 파일: [지침 대비 누락된 파일 목록 또는 없음]
git 상태: [Clean 또는 변경 상태 요약]
백업 필요 여부: [필요(경로 표기) / 불필요]
예상 수정 범위: [예: 시공사례 추가 및 index, cases 갱신]
```

## 04 승인 정책

- 안전한 작업은 사용자 승인 없이 자동 진행한다.
- 파일 읽기, 디렉터리 탐색, `git status`, `git diff` 확인은 자동 진행한다.
- HTML 생성, CSS 수정, JS 수정, `sitemap.xml` 갱신, `robots.txt` 갱신, `cases.html` 갱신, `index.html` 갱신, JSON-LD 생성, 이미지 최적화, 이미지 복사, 신규 폴더 생성도 자동 진행한다.
- `Remove-Item`, `git reset --hard`, `git clean -fd`, `git push -f`, `.git` 삭제, `images` 전체 삭제, `cases` 전체 삭제, `gallery` 전체 삭제, 프로젝트 구조 변경, 대량 파일 삭제는 반드시 사용자 승인을 받는다.
- 삭제가 필요한 경우에는 삭제 목록 출력, 삭제 이유 설명, 사용자 승인 순서로 진행한다.
- `git add`, `git commit`은 자동 진행할 수 있지만, `git push`는 뒤의 GitHub 자동 업데이트 규칙과 충돌 없이 진행한다.

## 05 백업 정책

- 여러 파일을 동시에 수정하거나 공통 CSS/JS를 건드릴 때는 백업을 수행한다.
- 백업 위치는 프로젝트 루트의 상위 폴더다.
- 백업 폴더명 형식은 `obaksa-home_backup_YYYYMMDD_HHMMSS`다.
- 백업 방식은 원본 구조 전체 복사 방식이다.
- 백업 제외 대상은 `.git`, `node_modules`, 기존 ZIP 파일, `*.bak`, `*.tmp`, `*.log`, `__MACOSX`, `.DS_Store`다.
- 백업이 완료되면 최종 보고서에 백업 폴더 절대 경로를 기록한다.
- 작업 도중 깨짐, 구문 오류, 파일 유실, 레이아웃 붕괴, 검증 실패가 발생하면 즉시 추가 작업을 중단한다.

## 06 Git 정책

- Git 충돌이 발생하면 임의 병합하지 않는다.
- 안전성이 검증된 수정사항만 `git status --short`, `git diff --name-only`, `git diff --stat`, `git add .`, `git commit`, `git push` 순서로 진행한다.
- 커밋 메시지는 짧고 명확하게 작성한다.
- `git push -f`, `git reset --hard`, `git clean -fd`, `git rebase --skip`, `git merge --strategy-option ours`, `git merge --strategy-option theirs`, `git pull --rebase`는 사용하지 않는다.
- 신규 시공사례 생성, 이미지 추가, `index.html` 갱신, `cases.html` 갱신, `sitemap.xml` 갱신이 포함된 작업은 GitHub 업데이트까지 자동으로 진행한다.
- 작업 완료 기준은 로컬 생성이 아니라 GitHub Pages 반영 준비 완료 상태다.
- `git status`에서 예상하지 못한 대량 변경, 승인되지 않은 삭제, merge conflict, push 권한 오류, 원격 주소 비정상, 요청 범위를 벗어난 변경이 있으면 자동 push를 중단하고 보고한다.

## 07 프로젝트 구조

- 루트 기준 주요 파일은 `index.html`, `about.html`, `cases.html`, `community.html`, `contact.html`, `channels.html`, `services.html`, `footer.html`, `assets/`, `images/`, `sitemap.xml`, `robots.txt`, `make-release.ps1`, `AGENTS.md`다.
- `README.md`와 `make-chatgpt-package.bat`은 존재할 때만 관리한다.
- `SITE_UPDATE_GUIDE.md`는 현재 루트에 존재하므로 참고 문서 및 보조 관리 문서로 유지한다.
- `images/cases/`와 `images/gallery/`는 핵심 이미지 자산이다.
- 경로 수정 후에는 반드시 실제 파일 존재 여부를 검증한다.

## 08 시공사례 자동 생성

- 신규 시공사례 제목은 `[지역명/동네명] + [문제 상황] + [해결 방법]`을 따른다.
- 블로그 URL이 제공되면 제목, 날짜, `slug`, 태그, 요약문을 자동 추출 또는 생성한다.
- 사용자가 별도로 제목, 날짜, `slug`, 태그, 요약문을 제공하지 않아도 된다.
- 신규 시공사례 HTML에는 `<article>` 최상위에 `data-title`, `data-date`, `data-category`, `data-tags`, `data-slug`, `data-summary`, `data-thumb`를 포함한다.
- `index.html`에는 최신 시공사례 1개만 노출한다.
- `cases.html` 상단 하이라이트 미리보기 카드는 최신 시공사례 2개만 노출한다.
- 신규 게시물이 추가되면 기존 1번은 2번으로 밀리고, 기존 2번은 하단 전체 리스트로 내려간다.
- 상세보기 버튼 문구는 `상세보기`로 통일한다.
- Instagram 링크 버튼 문구는 `Instagram`으로 통일한다.
- 모바일 초소형 버튼에서만 예외적으로 `보기`를 사용할 수 있다.

## 09 upload_images 규칙

- 작업 시작 시 반드시 `G:\OneDrive\01_울산오박사인테리어\obaksa_site\upload_images`를 확인한다.
- 이미지가 존재하면 신규 시공사례 후보로 간주한다.
- 자동 수행 순서는 이미지 확인, 신규 폴더 생성, `original` 폴더 생성, 원본 복사, 웹용 이미지 생성, `thumb` 생성이다.
- 새 시공사례 이미지는 `images/cases/[YYYY-MM-DD-영문작업명]/`에 저장한다.
- 폴더명과 파일명에는 한글, 공백, 대문자를 사용하지 않는다.
- 허용 문자는 영문 소문자, 숫자, 하이픈이다.
- 표준 파일명은 `thumb.jpg`, `before-01.jpg`, `before-02.jpg`, `after-01.jpg`, `after-02.jpg`, `instagram-thumb.jpg`, `webtoon-8cut.jpg`다.

## 10 이미지 최적화 규칙

- 리스트 썸네일은 `200KB ~ 400KB`를 권장한다.
- 일반 시공 사진은 `300KB ~ 800KB`를 권장한다.
- 메인 히어로/대표 이미지는 `800KB ~ 1.5MB`를 권장한다.
- 웹툰형 세로 장형 이미지는 `1MB ~ 2MB`를 권장한다.
- 파일명 정리, 경로 정리, 존재 여부 검증은 수행할 수 있다.
- 별도 압축 도구가 없으면 직접 압축이나 리사이징은 수행하지 말고 용량 초과 여부를 보고한다.
- 이미지가 깨지면 폴더명, 파일명, 확장자, 대소문자, 상대 경로, 실제 존재 여부를 순서대로 확인한다.

## 11 시공사례 정렬 규칙

- `cases.html` 갱신 시 모든 시공사례는 `data-date` 값 기준으로 내림차순 정렬한다.
- 최신 날짜가 항상 최상단에 위치해야 한다.
- 파일 생성 순서가 아니라 실제 시공 날짜를 기준으로 정렬한다.
- 신규 시공사례를 추가할 때는 기존 목록 전체를 다시 정렬한다.

---

## 12 콘텐츠 작성 규칙

- 문구는 고객 문의를 이끌어내는 영업 자산으로 작성한다.
- 현장의 신뢰감, 꼼꼼함, 안전한 마감, 생활 불편 해결을 전달한다.
- 시공사례 본문은 고객 문의 상황, 원인 분석, 작업 과정, 결과, 관리 팁, 문의 유도 흐름을 따른다.
- 식당, 카페, 캠핑장, 제조업 문체를 섞지 않는다.
- 고객 노출 영역에서는 `HTML 구조`, `카드형 구성`, `CSS`, `JavaScript`, `모바일 최적화`, `검색엔진`, `SEO`, `데이터`, `시스템`, `알고리즘`, `구현`, `자동화`, `플랫폼`, `렌더링`, `컴포넌트` 같은 개발자 용어를 쓰지 않는다.
- 과장 표현은 사용하지 않는다.

## 13 모바일 검증 규칙

- 모바일 검증은 실제 브라우저 렌더링 확인이 있어야 완료로 보고한다.
- 코드만 확인한 경우에는 `모바일 검증 완료`라고 보고하지 않는다.
- 검증 기준 폭은 `390px`, `430px`, `768px`다.
- 가로 스크롤, 텍스트 겹침, 버튼 잘림, 이미지 찌그러짐, 아코디언 이상, 플로팅 버튼 이상, 메뉴 이상, 카드 간격 과다, 좌우 여백 과다가 없어야 한다.
- 실제 브라우저 검증이 없으면 `모바일 검증 - 코드 검증 완료 / 실제 브라우저 검증 미실시`처럼 구분해 보고한다.

## 14 CSS 격리 규칙

- 새 기능 추가 시 반드시 고유 네임스페이스를 사용한다.
- 전역 선택자 수정은 최소화한다.
- `body`, `section`, `div`, `img`, `button`, `a`에 대한 광범위한 수정은 금지한다.
- 새 CSS는 기존 컴포넌트에 영향을 주지 않도록 상위 스코프를 명확히 둔다.
- 월드클락, 플로팅 버튼, 모바일 메뉴, 시공사례 카드, 히어로가 서로 충돌하지 않도록 스코프를 분리한다.

## 15 GitHub Pages 배포 규칙

- `git push` 완료 후 가능한 경우 GitHub Pages URL 접속 여부를 확인한다.
- 확인 대상은 메인 페이지, 시공사례 목록 페이지, 신규 시공사례 상세 페이지다.
- 배포 반영이 지연될 경우 `Git push는 완료되었습니다. GitHub Pages 반영은 인프라 상태에 따라 약간 지연될 수 있습니다.`로 보고한다.
- 신규 시공사례 작업은 신규 HTML 생성, 이미지 원본 `original` 보관, 웹용 이미지 생성, `index.html` 최신 사례 반영, `cases.html` 날짜순 정렬 반영, `sitemap.xml` 신규 URL 추가, JSON-LD 삽입, `git diff` 검증, `git add`, `git commit`, `git push`까지 끝나야 완료다.
- 자동 push 금지 조건은 승인 없는 삭제, `AGENTS.md`와 `SITE_UPDATE_GUIDE.md` 외 대량 문서 변경, `images/gallery` 전체 변경, `images/cases` 전체 일괄 변경, `.gitignore` 전체 교체, `package-lock`/`node_modules`/대용량 zip 포함, `git status` 충돌 상태다.

## 16 완료 보고서 규칙

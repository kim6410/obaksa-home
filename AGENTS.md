# AGENTS.md

## 0. 문서 목적

이 문서는 오박사만능인테리어 홈페이지를 Codex 또는 자동화 에이전트가 안정적으로 수정, 검증, 백업, 커밋, 동기화, 배포할 수 있도록 만든 최종 작업 지침서이다.

기존 `SITE_UPDATE_GUIDE.md`와 별도 `AGENTS.md`의 내용을 통합한 단일 기준 문서로 사용한다.

앞으로 오박사만능인테리어 프로젝트에서는 이 파일 하나를 최우선 기준으로 삼는다.

작업자는 HTML, CSS, JS, 이미지, 시공사례, sitemap, robots, 배포 ZIP, GitHub Pages 동기화 작업을 하기 전에 반드시 이 문서를 먼저 확인한다.

이 문서의 목표는 아래와 같다.

1. 기존에 잘 작동하는 화면 구성을 깨뜨리지 않는다.
2. 모바일 화면 품질을 유지한다.
3. 시공사례 추가 작업을 표준화한다.
4. 나중에 DB 게시판이나 CMS로 이전할 수 있도록 게시물 메타데이터를 유지한다.
5. 가능한 한 사용자 추가 컨펌 없이 자동으로 백업, 수정, 검증, 보고까지 진행한다.
6. 복구가 어려운 위험 작업만 사용자 확인을 받는다.
7. GitHub 저장소 동기화와 GitHub Pages 배포까지 가능한 경우 자동으로 진행한다.

## 1. 프로젝트 정체성

오박사만능인테리어 홈페이지는 울산 지역 기반의 생활밀착형 집수리, 욕실수리, 누수복구, 방화문수리, 조명교체, 도어락, 도배, 장판, 인테리어 시공 사례를 소개하고 문의 전환을 만드는 정적 HTML 홈페이지이다.

이 홈페이지는 단순한 포트폴리오가 아니라 고객 문의를 만드는 영업 자산이다.

핵심 가치는 아래와 같다.

1. 현장감
2. 신뢰감
3. 꼼꼼함
4. 안전성
5. 내구성
6. 생활 불편 해결
7. 30년 경력의 실무 감각
8. A/S와 사후관리 신뢰
9. 모바일에서 빠르게 확인 가능한 시공사례
10. 전화 또는 외부 채널로 이어지는 문의 전환

화려한 디자인보다 고객이 실제 사례를 보고 안심하는 것이 더 중요하다.

## 3. 기본 작업 원칙

- 먼저 실제 파일 내용을 확인한 뒤 수정한다.
- 추측으로 구조를 바꾸지 않는다.
- HTML 구조, 클래스명, 링크 대상은 가능한 그대로 유지한다.
- 필요한 경우에만 최소 범위로 수정한다.
- 한 번에 많이 바꾸기보다 검증 가능한 단위로 바꾼다.
- 기존에 잘 작동하던 모바일 레이아웃은 함부로 바꾸지 않는다.
- 요청받지 않은 대규모 리팩토링은 하지 않는다.
- 새 시공사례를 추가할 때는 `index.html`, `cases.html`, 개별 시공사례 HTML, 이미지 폴더, 필요 시 `sitemap.xml`까지 함께 점검한다.
- 고객이 실제 보는 영역에는 개발자 설명을 남기지 않는다.
- 작업 후에는 변경 파일, 검증 결과, 남은 리스크를 짧게 보고한다.

## 4. 자동 진행 원칙

이 프로젝트는 가능한 한 컨펌 없이 자동으로 진행한다.

사용자가 구체적으로 요청한 작업은 아래 절차에 따라 자동 진행한다.

```text
1. 현재 파일 확인
2. git status 확인
3. 필요한 경우 프로젝트 상위 폴더에 백업 생성
4. 요청 범위 수정
5. 링크, 이미지, 메타데이터, 한글 인코딩 검증
6. git diff 확인
7. 가능하면 commit
8. 가능하면 push
9. GitHub Pages 배포 반영 여부 확인
10. 작업 결과 보고
```

단, 복구가 어렵거나 대량 삭제가 포함된 작업은 사용자 확인을 받는다.

## 5. 자동 백업 규칙

여러 파일을 동시에 수정하거나, HTML/CSS/JS 구조에 영향을 주거나, 배포/동기화를 진행하기 전에는 현재 상태를 백업한다.

백업 위치는 반드시 프로젝트 상위 폴더로 한다.

프로젝트 기준 경로 예시는 아래와 같다.

```text
G:\OneDrive\01_울산오박사인테리어\obaksa_site\obaksa-home
```

상위 폴더 예시는 아래와 같다.

```text
G:\OneDrive\01_울산오박사인테리어\obaksa_site
```

백업 폴더명 형식은 아래를 따른다.

```text
obaksa-home_backup_YYYYMMDD_HHMMSS
```

예시:

```text
obaksa-home_backup_20260608_073000
```

백업 대상은 프로젝트 전체 폴더를 기본으로 하되, 아래 항목은 백업에서 제외할 수 있다.

```text
.git
node_modules
기존 ZIP
*.bak
*.tmp
*.log
__MACOSX
.DS_Store
```

백업을 만든 뒤에는 보고에 백업 폴더 경로를 포함한다.

삭제가 필요한 경우에는 먼저 삭제 후보 목록을 출력하고 사용자 확인을 받는다.

## 8. Git 동기화 및 배포 자동화 규칙

가능하면 수정 후 GitHub 저장소 동기화까지 자동으로 진행한다.

기본 순서는 아래와 같다.

```bash
git status --short
git diff --name-only
git diff --stat
git add .
git commit -m "Update obaksa site content and rules"
git push
```

커밋 메시지는 작업 내용에 맞게 짧고 명확하게 작성한다.

예시:

```text
Add obaksa case metadata rules
Update latest construction case
Fix mobile case card layout
Update sitemap for new case page
Clean duplicate floating assets
```

GitHub Pages 배포 확인은 가능한 경우 아래 방식으로 진행한다.

```text
1. git push 성공 여부 확인
2. GitHub Pages 주소 접속 확인
3. 주요 페이지 열림 확인
4. 새 시공사례 URL 열림 확인
5. 이미지와 CSS 반영 여부 확인
```

GitHub Pages 배포는 몇 분 지연될 수 있으므로, 즉시 반영되지 않으면 아래처럼 보고한다.

```text
Git push는 완료되었습니다.
GitHub Pages 배포 반영은 약간 지연될 수 있습니다.
현재 로컬 파일 기준 검증은 완료되었습니다.
```

원격 저장소 권한, 인증, 네트워크 문제로 push가 실패하면 무리하게 반복하지 말고 실패 사유를 보고한다.

## 9. 프로젝트 폴더 구조

기본 구조는 아래를 기준으로 한다.

```text
obaksa-home/
├─ index.html
├─ about.html
├─ cases.html
├─ community.html
├─ contact.html
├─ assets/
│  ├─ css/
│  └─ js/
├─ images/
│  ├─ cases/
│  └─ gallery/
├─ make-release.ps1
├─ make-chatgpt-package.bat
└─ AGENTS.md
```

### index.html

메인 랜딩 페이지이다.

최근 시공사례 1개가 반드시 노출되어야 한다.

문의 전환 버튼이 명확해야 한다.

모바일 아코디언 메뉴가 포함되어 있는 경우, 제목 옆에 펼침 표시가 있어야 한다.

### about.html

오박사만능인테리어 소개 페이지이다.

업체 소개, 신뢰 요소, 현장 경험, 서비스 철학, 작업 범위가 들어간다.

### cases.html

전체 시공사례 목록 페이지이다.

최근 시공사례 카드형 미리보기는 최근 2개만 노출한다.

아래쪽 목록에는 전체 시공사례가 리스트 형태로 표시된다.

### community.html

블로그, 인스타그램, 네이버 플레이스, 소식 콘텐츠 등 외부 채널과 연결되는 페이지이다.

### contact.html

전화, 위치, 상담 안내, 외부 링크가 들어가는 문의 페이지이다.

### channels.html

외부 채널, 소식, 인스타그램, 블로그 등 연결 정보를 정리하는 페이지이다.

### services.html

오박사만능인테리어의 주요 서비스와 작업 범위를 안내하는 페이지이다.

### footer.html

공통 푸터 영역의 문구, 링크, 연락 정보가 들어가는 공통 파일이다.

### assets/css/

전체 디자인, 반응형 레이아웃, 모바일 스타일을 관리한다.

### assets/js/

아코디언, 슬라이드, 인터랙션 등이 들어갈 수 있다.

### images/cases/

시공사례별 이미지 폴더이다.

### images/gallery/

기존 갤러리 이미지 폴더이다.

수정용 ZIP에는 기본적으로 포함하지 않지만, 실제 홈페이지에서 사용 중일 수 있으므로 함부로 삭제하지 않는다.

## 12. 인코딩 규칙

- 모든 HTML, CSS, JS, MD 파일은 UTF-8로 저장한다.
- 한글이 깨져 보이면 먼저 인코딩 문제를 의심한다.
- 저장 후에는 브라우저나 파일 읽기 결과에서 실제 한글 표시를 확인한다.
- `\r\n` 같은 문자 리터럴이 본문 텍스트로 들어가지 않도록 주의한다.
- JSON-LD 작성 시 스마트 따옴표가 들어가지 않도록 일반 따옴표를 사용한다.

## 13. 콘텐츠 문체 규칙

홈페이지의 모든 문구는 실제 방문 고객이 읽는다는 전제로 작성한다.

개발자 관점 설명은 고객 노출 문구에 쓰지 않는다.

고객이 겪는 불편과 해결 후 안심감을 먼저 말한다.

기술 설명은 고객이 이해할 수 있는 생활 언어로 바꾼다.

과장된 표현보다 실제 현장에서 확인한 내용을 우선한다.

문구 작성 우선순위는 아래 순서로 둔다.

```text
1. 고객이 겪은 문제 상황
2. 원인 분석
3. 현장에서 확인한 위험 요소
4. 시공 또는 수리 과정
5. 작업 후 달라진 점
6. 고객이 얻는 안심감
7. 재발 방지 또는 관리 팁
8. 문의 유도
```

식당, 카페, 캠핑장, 제조업 문체를 섞지 않는다.

오박사 콘텐츠는 아래 방향을 따른다.

```text
집수리
욕실수리
누수
곰팡이
결로
도배
장판
방화문
조명
수전
세면대
변기
도어락
현장 확인
안전 시공
꼼꼼한 마감
생활 불편 해결
A/S
```

```text
검색 노출과 모바일 가독성을 위해 HTML 카드형으로 구성했습니다.
```

올바른 표현:

```text
휴대폰에서도 시공 전후 사진과 작업 내용을 편하게 확인하실 수 있습니다.
```

올바른 표현:

잘못된 표현:

```text
슬라이드 구조를 개선했습니다.
```

올바른 표현:

## 16. 시공사례 제목 규칙

시공사례 제목은 아래 형식을 기본으로 한다.

```text
지역명 + 문제 상황 + 해결 방법
```

예시:

```text
울산 북구 신천동 세면대 붕괴 긴다리 세면대 교체
울산 북구 매곡동 LED 모듈 교체 전등 깜빡임 해결
울산 방화문 수리 닫히지 않던 현관문 복구
울산 북구 욕실 수전 누수 해바라기 샤워기 철거 후 일반 수전 교체
```

제목에 지역명, 문제, 해결 방법 중 하나가 빠지면 다시 검토한다.

```text
6. 작업 후 결과
7. 비슷한 상황의 고객에게 주는 조언
8. 문의 유도
```

## 19. cases.html 규칙

`cases.html` 상단 또는 중간의 카드형 미리보기는 최근 게시물 2개만 출력한다.

3개 이상 출력하지 않는다.

새 게시물을 추가하면 가장 최신 게시물이 1번 카드가 되고, 기존 최신 게시물이 2번 카드로 밀린다.

3번째 이후 게시물은 카드형 미리보기에서 제외하고 전체 목록에만 유지한다.

모바일 목록에서는 한 건이 지나치게 길어지지 않도록 한다.

모바일 목록 한 건의 기본 정보 구조는 아래처럼 유지한다.

```text
분류
날짜 + 작은 보기 버튼
제목
짧은 요약
```

## 21. 시공사례 article 메타데이터 필수 규칙

모든 개별 시공사례 HTML의 본문 최상위 `<article>`에는 DB 이전과 정적 검색을 위해 아래 `data-*` 속성을 반드시 포함한다.

```html
<article
  class="case-detail"
  data-title="울산 북구 세면대 교체"
  data-date="2026-06-05"
  data-category="욕실수리"
  data-tags="세면대교체,욕실수리,울산,신천동"
  data-slug="case-2026-06-05-sink-collapse"
  data-summary="신천동 무지개아파트에서 통째로 떨어진 세면대를 긴다리 세면대로 안전하게 교체한 시공사례입니다."
  data-thumb="images/cases/2026-06-05-sink-collapse/thumb.jpg"
>
```

필수 속성은 아래와 같다.

```text
data-title
data-date
data-category
data-tags
data-slug
data-summary
data-thumb
```

누락 시 작업 미완료로 간주한다.

### data-title

게시물 제목이다.

H1 제목과 같은 의미를 가져야 한다.

### data-date

작성일 또는 시공사례 등록일이다.

형식은 반드시 `YYYY-MM-DD`로 작성한다.

### data-category

시공 분류이다.

예시:

```text
욕실수리
누수복구
조명교체
방화문수리
도어락
도배장판
인테리어
집수리
기타수리
```

### data-tags

쉼표로 구분한다.

지역명, 동네명, 문제 상황, 작업 종류를 포함한다.

예시:

```text
울산,북구,신천동,세면대교체,욕실수리,긴다리세면대
```

### data-slug

파일명과 연결되는 고유 주소값이다.

영문 소문자, 숫자, 하이픈만 사용한다.

예시:

```text
case-2026-06-05-sink-collapse
```

### data-summary

검색 결과, 목록 카드, DB 이전 시 요약문으로 쓸 수 있는 80~160자 내외 문장이다.

### data-thumb

대표 이미지 경로이다.

이미지 파일이 실제 존재해야 한다.

## 22. JSON-LD 보강 규칙

가능하면 개별 시공사례 HTML에는 JSON-LD 구조화 데이터를 함께 넣는다.

단, 기존 구조를 크게 바꾸지 말고 최소 범위로 추가한다.

기본 예시는 아래와 같다.

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "울산 북구 세면대 교체, 신천동 무지개아파트 세면대 붕괴 복구",
  "datePublished": "2026-06-05",
  "author": {
    "@type": "Organization",
    "name": "오박사만능인테리어"
  },
  "publisher": {
    "@type": "Organization",
    "name": "오박사만능인테리어"
  },
  "image": "images/cases/2026-06-05-sink-collapse/thumb.jpg",
  "description": "신천동 무지개아파트에서 통째로 떨어진 세면대를 긴다리 세면대로 안전하게 교체한 시공사례입니다."
}
</script>
```

JSON-LD를 추가할 때는 한글 깨짐, 쉼표, 따옴표 오류를 반드시 확인한다.

## 23. 링크 및 버튼 규칙

블로그 링크가 있는 경우 `상세보기` 버튼에 연결한다.

인스타그램 릴스 링크가 있는 경우 `Instagram` 버튼에 연결한다.

카드형 미리보기 버튼명은 아래처럼 유지한다.

```text
상세보기
Instagram
```

아래 표현은 사용하지 않는다.

```text
자세히보기
자세히 보기
인스타릴스
인스타 릴스 보기
릴스보기
블로그 보기
더보기
```

단, 모바일 목록에서 아주 작은 pill 버튼이 필요한 경우에만 `보기`를 유지할 수 있다.

버튼 텍스트는 항상 잘 보여야 한다.

버튼 배경이 진하면 텍스트는 흰색을 유지한다.

호버 시에도 텍스트가 사라지거나 어두워지면 안 된다.

## 24. 이미지 관리 규칙

이미지는 홈페이지 용량을 가장 크게 늘리는 요소이다.

따라서 원본 이미지를 그대로 누적하지 않는다.

새 시공사례 이미지는 반드시 `images/cases/` 아래에 날짜와 작업 내용을 알 수 있는 폴더로 저장한다.

예시:

```text
images/cases/2026-06-07-led-module/
images/cases/2026-06-06-door-repair/
images/cases/2026-06-05-sink-collapse/
```

폴더명은 영어 소문자, 숫자, 하이픈을 사용한다.

한글 폴더명은 가능하면 사용하지 않는다.

공백은 사용하지 않는다.

이미지 파일명은 아래처럼 간결하게 작성한다.

```text
thumb.jpg
before-01.jpg
before-02.jpg
after-01.jpg
after-02.jpg
instagram-thumb.jpg
webtoon-8cut.jpg
```

파일명에 공백을 넣지 않는다.

카카오톡 원본 파일명처럼 긴 이름은 가능하면 정리한다.

홈페이지에는 압축본만 넣는다.

권장 용량은 아래와 같다.

```text
썸네일 이미지: 200KB ~ 400KB
일반 시공 사진: 300KB ~ 800KB
큰 대표 이미지: 800KB ~ 1.5MB
웹툰형 긴 이미지: 1MB ~ 2MB
```

원본 이미지는 OneDrive에 보관하고, 홈페이지에는 압축본만 사용한다.

## 26. 모바일 레이아웃 규칙

오박사만능인테리어 홈페이지는 모바일 방문자가 많을 것을 기준으로 한다.

모바일 화면에서는 정보가 높이를 너무 많이 차지하지 않도록 한다.

### 시공사례 목록 모바일 규칙

- 모바일에서 시공사례 목록 한 건은 최대한 컴팩트하게 표시한다.
- `보기` 버튼이 한 줄 전체를 차지하면 안 된다.
- 버튼은 날짜 옆에 작은 pill 버튼 형태로 배치한다.
- 분류는 날짜 위쪽에 배치해 목록 높이를 줄인다.

### 카드형 미리보기 모바일 규칙

- 모바일에서도 카드형 미리보기는 최근 2개만 노출한다.
- 카드 하나가 너무 길어지지 않도록 버튼 크기와 여백을 줄인다.
- 버튼은 가능한 한 한 줄에 배치한다.
- 공간이 부족하면 자연스럽게 줄바꿈은 가능하지만 버튼이 과도하게 커지면 안 된다.

### 아코디언 모바일 규칙

- 모바일 `index.html`의 아코디언 메뉴는 누를 수 있다는 표시가 반드시 있어야 한다.
- 제목 옆에는 기본 상태에서 아래 화살표를 표시한다.
- 펼쳐진 상태에서는 위 화살표를 표시한다.

```text
기본 상태: ▼
펼침 상태: ▲
```

### 모바일 여백 규칙

- 모바일에서는 좌우 여백이 과도하게 크면 안 된다.
- 이미지나 카드가 너무 작아 보이면 안 된다.
- 가로 스크롤이 생기면 안 된다.
- 버튼, 카드, 제목의 세로 간격은 지나치게 넓지 않게 유지한다.

## 27. HTML 수정 규칙

- 텍스트 수정과 구조 수정은 구분한다.
- 요청이 텍스트 수정이라면 문구만 바꾸고 태그 구조는 그대로 둔다.
- 닫는 태그 오류가 보이면 해당 블록 범위 안에서만 바로잡는다.
- 공통 네비게이션은 전 페이지에서 동일한 항목과 순서를 유지한다.
- 새 클래스명은 만들기 전에 기존 클래스가 이미 있는지 먼저 확인한다.
- 시공사례 상세 페이지에는 `<h1>`, `<article>`, 대표 이미지, 본문 소제목, 문의 유도 영역이 유지되어야 한다.
- 개별 시공사례 HTML을 추가할 때는 article `data-*` 메타데이터를 반드시 넣는다.

## 28. CSS 수정 규칙

- CSS는 필요한 경우에만 수정한다.
- 모바일 수정은 기본적으로 `@media screen and (max-width: 980px)` 기준으로 검토한다.
- PC 디자인을 건드리지 않는 것을 기본 원칙으로 한다.
- 히어로, 오버레이, 타이틀 크기, CTA 여백은 모바일에서 우선 확인한다.
- 중복된 선택자나 `!important`가 있으면 실제 우선순위를 확인하고 수정한다.
- 모바일에서 가로 스크롤이 생기지 않도록 한다.
- 버튼 텍스트는 배경색과 대비가 충분해야 한다.

## 29. 주요 클래스 확인 규칙

실제 클래스명은 파일마다 다를 수 있으므로 수정 전 반드시 `index.html`, `cases.html`, `assets/css/`에서 현재 사용 중인 클래스명을 확인한다.

다른 프로젝트에서 사용하던 클래스명은 오박사 프로젝트의 기준 클래스가 아니다.

오박사 프로젝트에서는 기존에 사용 중인 아래 계열 클래스명을 우선 유지한다.

```text
ob-
obaksa-
case-
cases-
```

새 클래스가 필요할 때는 가능하면 위 접두사를 사용한다.

## 30. 배포 ZIP 생성 규칙

배포용 ZIP과 수정용 ZIP은 목적이 다르다.

두 파일을 혼동하지 않는다.

### 배포용 ZIP

배포용 ZIP은 실제 GitHub Pages 또는 호스팅에 올릴 파일이다.

`make-release.ps1`이 있으면 이 스크립트를 우선 사용한다.

배포용 ZIP에는 홈페이지 작동에 필요한 파일이 들어가야 한다.

하지만 아래 항목은 절대 포함하지 않는다.

```text
.git/
.gitignore
.vscode/
node_modules/
*.bak
*.tmp
*.log
__MACOSX/
.DS_Store
기존 release zip
```

배포용 ZIP 파일명 형식은 아래를 따른다.

```text
obaksa-home_release_YYYYMMDD_HHMM.zip
```

### ChatGPT 수정용 ZIP

수정용 ZIP은 수정에 필요한 최소 파일만 담는다.

`make-chatgpt-package.bat`가 있으면 이 스크립트를 우선 사용한다.

수정용 ZIP 파일명 형식은 아래를 따른다.

```text
obaksa-home_chatgpt_YYYYMMDD_HHMM.zip
```

수정용 ZIP에는 아래 항목을 포함한다.

```text
루트 HTML 파일
assets/css/
assets/js/
cases.html 관련 파일
images/cases 최신 1~2개 폴더
AGENTS.md
```

수정용 ZIP에는 아래 항목을 제외한다.

```text
.git/
.vscode/
node_modules/
images/gallery/
오래된 images/cases 전체
*.zip
*.bak
*.tmp
*.log
```

## 31. 삭제하면 안 되는 파일과 폴더

아래 파일과 폴더는 함부로 삭제하지 않는다.

```text
.git/
.gitignore
index.html
about.html
cases.html
community.html
contact.html
assets/
assets/css/
assets/js/
images/
images/cases/
images/gallery/
make-release.ps1
make-chatgpt-package.bat
AGENTS.md
```

### 삭제 가능 후보

아래 항목은 확인 후 삭제할 수 있다.

```text
오래된 obaksa-home_release_*.zip
오래된 obaksa-home_chatgpt_*.zip
*.bak
*.tmp
*.log
.DS_Store
__MACOSX
```

단, 삭제 전에는 목록을 먼저 출력하고 확인 후 삭제한다.

## 35. 금지 사항

아래 작업은 하지 않는다.


## 36. 예외 처리

- 파일이 깨져 보이면 먼저 인코딩과 실제 저장 상태를 확인한다.
- 동일한 문제가 여러 페이지에 반복되면 공통 원인을 먼저 찾는다.
- 한 페이지만 고쳤는데 다른 페이지가 깨졌다면 공통 헤더/CSS를 먼저 의심한다.
- 이미지가 깨지면 폴더명, 파일명, 대소문자, 확장자, 상대경로를 먼저 확인한다.
- 시공사례 목록은 정상인데 상세 페이지가 깨지면 해당 개별 HTML의 head, CSS 링크, 이미지 상대경로를 먼저 확인한다.
- Git push가 실패하면 인증, 원격 저장소, 브랜치 상태를 확인하고 실패 사유를 보고한다.
- GitHub Pages가 즉시 반영되지 않으면 배포 지연 가능성을 보고한다.

## 37. Codex 실행용 기본 프롬프트

Codex에게 작업을 맡길 때는 아래 프롬프트를 기본으로 사용한다.

```text
프로젝트 루트의 AGENTS.md를 먼저 읽고 모든 작업 기준으로 삼아줘.

작업 전 git status --short를 확인하고, 여러 파일을 수정하거나 배포/동기화가 포함되면 프로젝트 상위 폴더에 obaksa-home_backup_YYYYMMDD_HHMMSS 형식으로 백업을 먼저 만들어줘.

요청받은 범위는 사용자 추가 컨펌 없이 자동으로 진행해줘.

단, git reset --hard, git clean -fd, .git 삭제, 프로젝트 폴더 삭제, 대량 파일 삭제, images/gallery 삭제, images/cases 전체 삭제, 외부 API 키 삭제, 환경설정 파일 전체 교체는 절대 자동 진행하지 말고 먼저 확인을 받아줘.

수정 후에는 한글 깨짐, 이미지 경로, 링크, 모바일 화면, article data-* 메타데이터, sitemap 반영 여부를 검증해줘.

가능하면 git add, commit, push까지 진행하고 GitHub Pages 배포 반영 여부도 확인해줘.

작업 완료 후에는 백업 폴더 경로, 수정 파일 목록, 삭제 파일 목록, git diff --name-only, git diff --stat, 핵심 변경 내용, Git commit 여부, Git push 여부, 배포 확인 결과, 남은 리스크, 되돌리는 방법을 짧게 보고해줘.
```

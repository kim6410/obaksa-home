# AGENTS.md

오박사만능인테리어 정적 홈페이지 운영 및 유지보수 마스터 가이드

버전: v3.0

적용 대상: `obaksa-home` 프로젝트

최종 목적: 기존 홈페이지를 깨뜨리지 않고, 시공사례와 콘텐츠를 안정적으로 누적하며, 모바일 방문자가 편하게 문의할 수 있는 구조를 유지한다.

---

## 1. 프로젝트 운영 철학

이 프로젝트의 최우선 목표는 "새 기능 추가"가 아니라 "기존 홈페이지를 절대 깨뜨리지 않는 것"이다.

안정성이 기능보다 우선한다.

수정 범위는 최소화한다.

기존 동작은 반드시 유지한다.

사용자가 요청하지 않은 구조 변경, 디자인 리뉴얼, 대규모 리팩토링은 금지한다.

작은 수정 하나도 실제 고객 문의와 연결되는 사업 자산을 다루는 일로 간주한다.

---

## 2. 최상위 규칙 및 우선순위

`AGENTS.md`는 이 프로젝트의 절대적인 마스터 가이드라인이다.

다른 문서, 예를 들어 `README.md`나 작업 메모와 내용이 충돌할 경우 반드시 `AGENTS.md`를 최우선 적용한다.

`SITE_UPDATE_GUIDE.md`는 현재 삭제 상태이므로 참고 문서로 사용하지 않는다.

새로운 파일이 추가되거나 파일 구조, 빌드 프로세스, 배포 방식이 변경될 경우 반드시 `AGENTS.md`를 먼저 갱신한 뒤 다음 작업을 진행한다.

`AGENTS.md`에 기술된 구조 및 규칙과 실제 파일 구조가 다를 경우 작업을 시작하지 말고 차이점을 사용자에게 즉시 보고한다.

에이전트가 이 문서를 완전히 읽지 않았거나 실제 구조와의 정합성을 검증하지 않았다면 어떠한 작업도 시작할 수 없다.

## 3. 작업 시작 전 필수 절차

모든 작업은 아래 순서를 따른다.

단계를 건너뛴 경우 최종 보고서에 반드시 그 사유를 명시한다.

1. `AGENTS.md` 전체 읽기
2. `git status --short` 실행
3. 프로젝트 루트 파일 및 폴더 구조 확인
4. `AGENTS.md`에 정의된 주요 파일 목록과 실제 파일 목록 비교
5. 누락 파일, 신규 파일, 비정상 구조 변경 발견 시 선보고
6. 다중 파일 수정, 공통 CSS/JS 수정, 배포/동기화 작업이 포함될 경우 상위 폴더에 원본 전체 복사 백업 생성
7. 작업 착수 전 아래 형식으로 단문 보고

```text
[AGENTS 확인 완료]

프로젝트: 오박사만능인테리어 정적 홈
파일 수: [확인된 파일 개수]
추가 파일: [신규 추가된 파일 목록 또는 없음]
누락 파일: [지침 대비 누락된 파일 목록 또는 없음]
git 상태: [Clean 또는 변경 상태 요약]
백업 필요 여부: [필요(경로 표기) / 불필요]
예상 수정 범위: [예: 시공사례 추가 및 index, cases 갱신]
```

## 4. 자동 작업 실행 순서

모든 자동화 작업은 아래 기본 루틴을 따른다.

```text
[AGENTS 확인]
→ [Git 상태 확인]
→ [구조 비교]
→ [상위 폴더 백업 생성]
→ [작업 수행]
→ [수정 파일 검증]
→ [Git Add]
→ [Git Commit]
→ [Git Push]
→ [배포 확인]
→ [최종 보고]
```

세부 확인 항목은 다음과 같다.

```bash
git status --short
git diff --name-only
git diff --stat
```

필요한 경우에만 다음을 수행한다.

```bash
git add .
git commit -m "[의미 있는 단문 메시지]"
git push
```

권한 오류, 인증 실패, 네트워크 단절, 충돌 발생 시 무리하게 재시도하지 말고 즉시 실패 사유를 보고한다.

## 5. 최소 수정 원칙

사용자가 요청한 범위를 벗어난 수정은 금지한다.

예시는 다음과 같다.

```text
전화번호 수정
→ 연락처 관련 파일만 수정

시공사례 추가
→ 사례 관련 파일, index.html, cases.html, sitemap.xml만 필요 범위 내 수정

이미지 교체
→ 이미지 파일과 해당 이미지 경로만 수정

오타 수정
→ 해당 문구만 수정
```

사용자가 요청하지 않은 아래 작업은 금지한다.

- 전체 CSS 재구성
- 전체 HTML 구조 변경
- 전체 JS 리팩토링
- PC 디자인 리뉴얼
- 전역 레이아웃 재설계
- 불필요한 라이브러리 추가

수정 전에는 반드시 영향 범위를 추정한다.

```text
수정 대상: footer.html
영향 범위: 공통 하단, 연락처, 전체 페이지 하단 표시
예상 영향 페이지: index.html, about.html, cases.html, contact.html 등
```

## 6. 자동 승인 및 사용자 확인 정책

에이전트는 사용자의 피로도를 줄이기 위해 안전 범위 내의 작업은 자동 진행할 수 있다.

단, 파괴적이거나 복구가 어렵거나 사업 자산에 손상을 줄 수 있는 작업은 반드시 중단하고 사용자 승인을 받아야 한다.

### 6.1 사용자 확인 없이 자동 진행 가능한 작업

- 파일 읽기
- 디렉터리 탐색
- `git status` 확인
- `git diff` 추적
- 상위 폴더 백업 생성
- 배포용 또는 ChatGPT 수정용 ZIP 패키징
- HTML 구조를 유지한 문구 수정
- CSS/JS 부분 보정
- 오타, 링크, 이미지 경로 수정
- 시공사례 추가
- `data-*` 메타데이터 보강
- 고객 친화적 문구 개선
- 모바일 레이아웃 밀림 보정
- `sitemap.xml` 갱신
- `robots.txt` 보정
- `README.md`는 존재할 때만 관리
- `make-chatgpt-package.bat`은 존재할 때만 관리
- `AGENTS.md` 자가 갱신
- 중복 코드 제거
- 공통 컴포넌트의 명백한 오류 수정
- 검증 완료 후 일반적인 `git add`, `git commit`, `git push`

### 6.2 사용자 확인이 필요한 작업

아래 작업은 반드시 사용자 승인 후 진행한다.

- `git reset --hard`
- `git clean -fd`
- 강제 푸시
- `git push -f`
- `git pull --rebase`
- `.git` 폴더 삭제
- 원격 저장소 주소 변경
- 브랜치 삭제
- GitHub 저장소 삭제
- 프로젝트 전체 폴더 삭제
- 상위 폴더 삭제
- 드라이브 루트 삭제
- `images/cases` 전체 삭제
- `images/gallery` 전체 삭제
- 파일 또는 폴더 대량 삭제
- 외부 API 키 삭제
- 환경설정 파일 전체 교체
- `.gitignore` 전체 교체 또는 위험 수정

삭제가 필요한 경우 아래 순서를 지킨다.

1. 삭제 후보 파일과 폴더 목록 출력
2. 삭제 이유 설명
3. 복구 가능 여부 설명
4. 사용자 서면 승인 획득
5. 승인 후 백업 생성
6. 삭제 진행

## 7. 백업 및 롤백 규칙

### 7.1 백업 원칙

여러 파일을 동시에 수정하거나 공통 CSS/JS를 건드릴 때, 혹은 Git 배포를 실행하기 전에는 반드시 백업을 수행한다.

백업 위치는 프로젝트 루트의 상위 폴더이다.

프로젝트 경로 예시는 다음과 같다.

```text
G:\OneDrive\01_울산오박사인테리어\obaksa_site\obaksa-home
```

백업 생성 경로 예시는 다음과 같다.

```text
G:\OneDrive\01_울산오박사인테리어\obaksa_site\
```

백업 폴더명은 다음 형식을 따른다.

```text
obaksa-home_backup_YYYYMMDD_HHMMSS
```

예시는 다음과 같다.

```text
obaksa-home_backup_20260608_074000
```

백업 방식은 원본 구조 전체 복사 방식이다.

Windows 예시:

```powershell
Copy-Item -Path "obaksa-home" -Destination "obaksa-home_backup_YYYYMMDD_HHMMSS" -Recurse
```

macOS/Linux 예시:

```bash
cp -R obaksa-home obaksa-home_backup_YYYYMMDD_HHMMSS
```

백업 제외 가능 대상은 다음과 같다.

```text
.git
node_modules
기존 ZIP 파일
*.bak
*.tmp
*.log
__MACOSX
.DS_Store
```

백업이 완료되면 최종 보고서에 백업 폴더의 절대 경로를 기록한다.

### 7.2 치명적 오류 발생 시 롤백 프로토콜

작업 도중 깨짐, 구문 오류, 파일 유실, 레이아웃 붕괴, 검증 실패가 발생하면 즉시 추가 작업을 중단한다.

사용자에게 최신 백업 폴더 경로를 보고한다.

아래와 같은 복구 절차를 제공한다.

Windows 예시:

```powershell
Rename-Item "obaksa-home" "obaksa-home_broken_YYYYMMDD_HHMMSS"
Copy-Item -Path "obaksa-home_backup_YYYYMMDD_HHMMSS" -Destination "obaksa-home" -Recurse
```

macOS/Linux 예시:

```bash
mv obaksa-home obaksa-home_broken_YYYYMMDD_HHMMSS
cp -R obaksa-home_backup_YYYYMMDD_HHMMSS obaksa-home
```

## 8. Git 충돌 및 원격 동기화 규칙

원격 저장소와의 동기화 과정에서 충돌이 발생하면 에이전트 임의로 병합하지 않는다.

충돌 즉시 작업을 중단하고 충돌 파일 목록과 사유를 사용자에게 보고한다.

절대 사용 금지 명령어는 다음과 같다.

```bash
git push -f
git reset --hard
git clean -fd
git rebase --skip
git merge --strategy-option ours
git merge --strategy-option theirs
git pull --rebase
```

안전성이 검증된 수정사항만 아래 흐름으로 동기화한다.

```bash
git status --short
git diff --name-only
git diff --stat
git add .
git commit -m "[의미 있는 단문 메시지]"
git push
```

커밋 메시지는 짧고 명확하게 작성한다.

좋은 예시는 다음과 같다.

```text
Update latest construction case
Fix mobile case card layout
최신 욕실수리 시공사례 추가 및 시트맵 갱신
```

Push 완료 후 가능한 환경이라면 GitHub Pages URL 접속을 확인한다.

반영 지연 시 아래 문구를 사용한다.

```text
Git push는 완료되었습니다. GitHub Pages 배포 반영은 인프라 환경에 따라 약간 지연될 수 있습니다. 현재 로컬 파일 기준 검증은 완료되었습니다.
```

## 9. 프로젝트 폴더 구조 및 파일별 정의

홈페이지 프로젝트는 아래의 정적 구조를 표준으로 유지한다.

Windows 로컬 경로는 백슬래시를 사용한다.

웹 URL과 상대 경로는 슬래시를 사용한다.

경로 수정 후에는 반드시 실제 파일 존재 여부를 검증한다.

```text
obaksa-home/
├─ index.html                    # 메인 랜딩 페이지, 최근 시공사례 1개 필수 노출
├─ about.html                    # 업체 소개, 30년 경력 신뢰 요소 및 서비스 철학
├─ cases.html                    # 시공사례 목록, 상단 최근 카드 2개 고정
├─ community.html                # 블로그, 인스타그램, 네이버 플레이스 등 외부 채널 연결
├─ contact.html                  # 전화번호, 상세 위치, 상담 시간 및 문의 안내
├─ channels.html                 # 외부 소셜 채널 및 소식처 정보 정리 페이지
├─ services.html                 # 주요 서비스 범위 및 시공 단가 가이드
├─ footer.html                   # 공통 하단 영역
├─ assets/
│  ├─ css/                       # 전체 스타일시트 및 모바일 반응형 관리
│  └─ js/                        # 아코디언, 모바일 메뉴, 인터랙션 스크립트
├─ images/
│  ├─ cases/                     # 시공사례별 이미지 폴더
│  └─ gallery/                   # 기존 정적 갤러리 자산, 무단 삭제 금지
├─ sitemap.xml                   # 검색 수집용 사이트맵
├─ robots.txt                    # 크롤러 접근 제어
├─ make-release.ps1              # 배포용 PowerShell 빌드 스크립트
├─ make-chatgpt-package.bat      # 존재할 때만 관리하는 ChatGPT 수정용 패키징 스크립트
└─ AGENTS.md                     # 본 마스터 지침 문서
```

실제 파일 구조가 위 구조와 다를 경우, 차이점을 먼저 보고하고 필요 시 `AGENTS.md` 갱신 여부를 판단한다.

`README.md`는 현재 루트에 없으므로 필수 파일이 아니다. 존재할 때만 관리한다.

`make-chatgpt-package.bat`도 현재 루트에 없으므로 필수 파일이 아니다. 존재할 때만 관리한다.

`SITE_UPDATE_GUIDE.md`는 현재 삭제 상태이므로 루트 기준 관리 파일에서 제외한다.

## 10. 콘텐츠 문체 및 마케팅 자산화 규칙

홈페이지의 모든 문구는 단순한 텍스트가 아니라 고객의 전화를 이끌어내는 마케팅 영업 자산이다.

기술적 나열보다 현장의 신뢰감, 꼼꼼함, 안전한 마감, 생활 불편 해결을 전달하는 데 집중한다.

### 10.1 콘텐츠 전개 표준 8단계

시공사례와 본문 콘텐츠는 아래 흐름을 기본으로 작성한다.

1. 현장 위치와 고객의 최초 문의 상황 소개
2. 고객이 일상에서 느꼈던 불편함 또는 안전 위험 설명
3. 30년 경력의 안목으로 현장에서 진단한 원인 분석
4. 시공 전 안전을 위해 확인한 포인트 명시
5. 실제 시공 및 수리 과정 기록
6. 시공 완료 후 전후 결과 설명
7. 재발 방지 관리 팁 제공
8. 자연스러운 전화 문의 및 외부 상담 채널 유도

### 10.2 업종 아이덴티티 유지

식당, 카페, 캠핑장, 제조업 등 타 업종 문체를 섞지 않는다.

아래 키워드를 중심으로 생활밀착형 인테리어 전문 문체를 유지한다.

## 11. 시공사례 작성 및 데이터 자산화 규칙

### 11.1 시공사례 제목 공식

시공사례 제목은 아래 공식을 따른다.

```text
[지역명/동네명] + [고객의 문제 상황] + [명확한 해결 방법]
```

올바른 예시는 다음과 같다.

```text
울산 북구 신천동 세면대 붕괴 긴다리 세면대 교체
울산 북구 매곡동 LED 모듈 교체 전등 깜빡임 해결
울산 방화문 수리 닫히지 않던 현관문 복구
```

### 11.2 시공사례 7대 메타데이터 필수 규칙

향후 정적 페이지에서 DB 기반 CMS나 게시판 시스템으로 이전할 때 자산 유실을 막기 위해 모든 개별 시공사례 HTML의 최상위 `<article>` 태그에는 아래 `data-*` 속성 7가지를 포함한다.

하나라도 누락될 경우 배포 완료로 간주하지 않는다.

```html
<article
  class="case-detail"
  data-title="울산 북구 신천동 세면대 붕괴 긴다리 세면대 교체"
  data-date="2026-06-05"
  data-category="욕실수리"
  data-tags="울산,북구,신천동,세면대교체,욕실수리,긴다리세면대"
  data-slug="case-2026-06-05-sink-collapse"
  data-summary="신천동 무지개아파트에서 통째로 떨어진 세면대를 긴다리 세면대로 안전하게 교체 시공한 사례입니다."
  data-thumb="images/cases/2026-06-05-sink-collapse/thumb.jpg"
>
```

각 속성의 정의는 다음과 같다.

```text
data-title   : 시공사례 최종 제목, h1 텍스트와 일치
data-date    : 작성일 또는 시공 완료일, YYYY-MM-DD 형식
data-category: 지정 분류 중 하나
data-tags    : 쉼표로 구분, 공백 없이 작성
data-slug    : 영문 소문자, 숫자, 하이픈만 허용
data-summary : 80자~160자 내외 요약문
data-thumb   : 실제 존재하는 대표 이미지 경로
```

분류 풀은 다음과 같다.

```text
욕실수리
누수복구
조명교체
방화문수리
도어락
도배장판
인테리어
집수리
기타수리
```

### 11.3 시공사례 노출 규칙

`index.html`에는 최신 시공사례 1개만 하이라이트로 노출한다.

`cases.html` 상단 하이라이트 미리보기 카드는 최신 시공사례 2개만 노출한다.

신규 게시물이 추가되면 기존 1번은 2번으로 밀리고, 기존 2번은 하단 전체 리스트로 내려간다.

상세보기 버튼 문구는 `상세보기`로 통일한다.

Instagram 링크 버튼 문구는 `Instagram`으로 통일한다.

모바일 초소형 버튼에서만 예외적으로 `보기`를 사용할 수 있다.

금지 버튼 문구는 다음과 같다.

```text

자세히 보기
인스타릴스
인스타 릴스 보기
릴스보기
블로그 보기
더보기
```

## 12. 검색 수집 보강 및 구조화 데이터 규칙

개별 시공사례 문서에는 최소 범위의 JSON-LD 구조화 데이터를 `<head>` 영역 또는 `<article>` 직전에 삽입한다.

작성 시 스마트 따옴표를 사용하지 않는다.

반드시 일반 반각 따옴표를 사용한다.

`@context` 값은 아래처럼 정확히 작성한다.

```json
"@context": "https://schema.org"
```

잘못된 예시는 다음과 같다.

```json
"@context": "[https://schema.org](https://schema.org)"
```

올바른 예시는 다음과 같다.

```html
<script type="application/ld+json">
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "울산 북구 세면대 교체, 신천동 무지개아파트 세면대 붕괴 복구",
  "datePublished": "2026-06-05",
  "author": {
    "@type": "Organization",
    "name": "오박사만능인테리어"
  },
  "publisher": {
    "@type": "Organization",
    "name": "오박사만능인테리어"
  },
  "image": "images/cases/2026-06-05-sink-collapse/thumb.jpg",
  "description": "신천동 무지개아파트에서 통째로 떨어진 세면대를 긴다리 세면대로 안전하게 교체 시공한 사례입니다."
}
</script>
```

새로운 시공사례 주소가 추가되면 `sitemap.xml` 반영 여부를 확인한다.

`robots.txt` 변경은 필요한 경우에만 수행한다.

## 13. 모바일 우선 및 UI 레이아웃 규칙

방문자의 다수가 스마트폰 유입이므로 모바일 레이아웃의 가독성은 핵심 품질 지표이다.

모든 수정은 PC보다 모바일을 먼저 검증한다.

검증 기준 폭은 다음과 같다.

```text
390px
430px
768px
```

필수 확인 항목은 다음과 같다.

- 가로 스크롤 발생 여부
- 텍스트 겹침 여부
- 버튼 잘림 여부
- 이미지 찌그러짐 여부
- 아코디언 정상 동작 여부
- 플로팅 버튼 정상 표시 여부
- 메뉴 정상 표시 여부
- 카드 간격 과다 여부
- 좌우 여백 과다 여부

스타일 수정 시 PC 디자인을 훼손하지 않도록 모바일 전용 수정은 가급적 아래 조건절 안에서 처리한다.

```css
@media screen and (max-width: 980px) {
  /* 모바일 전용 조정 */
}
```

### 13.1 시공사례 목록 모바일 컴팩트 규칙

`cases.html`의 목록은 세로 스크롤 압박을 줄인다.

상세 링크 버튼이 가로 한 줄 전체를 차지하여 공간을 낭비하지 않게 한다.

날짜 우측 또는 정렬 영역에 작은 알약 모양 버튼으로 배치할 수 있다.

카테고리 뱃지는 날짜 위쪽에 배치하여 가로폭을 확보한다.

상세보기와 Instagram 버튼은 가급적 가로 한 줄 균등 분할을 지향한다.

### 13.2 모바일 아코디언 규칙

모바일 메뉴, FAQ, 서비스 목록 등 접고 펼치는 요소에는 우측 끝에 상태 표시를 둔다.

```text
접힌 상태: ▼
펼쳐진 상태: ▲
```

사용자가 누를 수 있는 영역임을 명확히 보여야 한다.

### 13.3 스크롤 및 여백 통제

모바일에서 가로 스크롤이 발생하면 수정 완료로 간주하지 않는다.

모바일 좌우 패딩은 과도하게 넓히지 않는다.

기본 안전 여백은 15px 내외를 기준으로 한다.

텍스트와 버튼의 색상 대비를 명확히 한다.

야외에서 보아도 버튼 글자가 묻히지 않게 한다.

## 14. CSS 격리 및 충돌 방지 규칙

오박사 프로젝트는 월드클락, 플로팅 버튼, FAQ, 히어로 슬라이드, 시공사례 카드처럼 서로 다른 컴포넌트가 함께 존재한다.

새 기능 추가 시 반드시 고유 네임스페이스를 사용한다.

금지 예시는 다음과 같다.

```css
.card {}
.title {}
.menu {}
.container {}
.button {}
```

권장 예시는 다음과 같다.

```css
.ob-clock {}
.ob-faq {}
.ob-floating {}
.ob-case {}
.ob-service {}
.ob-hero {}
```

전역 선택자 수정은 최소화한다.

아래 선택자에 대한 광범위한 수정은 금지한다.

```css
body {}
section {}
div {}
img {}
button {}
a {}
```

부득이하게 전역 선택자를 수정해야 할 경우 영향 범위를 먼저 보고한다.

새 CSS는 기존 컴포넌트에 영향을 주지 않도록 상위 스코프를 명확히 둔다.

예시는 다음과 같다.

```css
.ob-case-list .ob-case-card {}
.ob-floating-wrap .ob-floating-btn {}
.ob-clock-widget .ob-clock-city {}
```

월드클락, 플로팅 버튼, 모바일 메뉴, 시공분야 섹션은 서로 영향을 주지 않도록 독립 스코프를 유지한다.

## 16. 이미지 관리 및 경로 검증 규칙

### 16.1 핵심 이미지 자산 보호

다음 경로는 핵심 사업 자산이다.

```text
images/cases/

```

사용자 승인 없이 삭제, 이동, 이름 변경을 금지한다.

### 16.2 디렉터리 분리 체계

모든 새 시공사례 이미지는 아래 경로에 저장한다.

```text
images/cases/[YYYY-MM-DD-영문작업명]/
```

올바른 예시는 다음과 같다.

```text
images/cases/2026-06-07-led-module/
```

폴더명과 파일명에는 한글, 공백, 대문자를 사용하지 않는다.

허용 문자는 영문 소문자, 숫자, 하이픈이다.

### 16.3 표준 파일명 규칙

```text
thumb.jpg
before-01.jpg
before-02.jpg
after-01.jpg
after-02.jpg
instagram-thumb.jpg
webtoon-8cut.jpg
```

### 16.4 이미지 용량 권장 범위

```text
리스트 썸네일 이미지: 200KB ~ 400KB
일반 시공 프로세스 사진: 300KB ~ 800KB
메인 히어로/대표 이미지: 800KB ~ 1.5MB
웹툰형 세로 장형 이미지: 1MB ~ 2MB
```

에이전트는 파일명 정리, 경로 정리, 존재 여부 검증은 수행할 수 있다.

로컬 환경에 FFmpeg, ImageMagick 등 별도 도구가 없으면 직접 압축이나 리사이징은 수행하지 말고 사용자에게 용량 초과 여부를 보고한다.

### 16.5 이미지 엑스박스 오류 점검

이미지가 깨지면 아래 항목을 순서대로 확인한다.

1. 폴더명 오타
2. 파일명 오타
3. 확장자 불일치
4. 대소문자 불일치
5. 상대 경로 오류
6. 파일 실제 존재 여부

## 19. 금지 사항 및 문제 해결 규칙

### 19.1 절대 금지 사항

- 코드 문맥 확인 없는 Replace All
- 전체 CSS 재작성
- 전체 HTML 재작성
- 전체 JS 재작성
- 글로벌 네비게이션 구조 무단 변경
- 푸터 구조 무단 변경
- 히어로 섹션 클래스 체계 무단 변경
- 사용자 요청 범위를 벗어난 PC 버전 리팩토링
- `data-*` 필수 메타데이터 없이 시공사례 HTML만 추가
- 타 업종 문구를 오박사 페이지에 방치

### 19.2 한글 깨짐 대응

한글이 깨지면 즉시 저장 포맷을 UTF-8로 재지정한다.

`
` 같은 개행 리터럴이 화면에 그대로 출력되는지 확인한다.

### 19.3 레이아웃 전체 파괴 대응

한 페이지만 수정했는데 여러 페이지가 깨진 경우 공통 CSS, 공통 JS, footer, header, 닫는 태그 누락을 먼저 의심한다.

수정 범위를 되짚고 `git diff`로 예상 밖 변경이 있는지 확인한다.

### 19.4 링크 오류 대응

링크 오류는 아래 순서로 확인한다.

1. 상대 경로 확인
2. 파일 존재 여부 확인
3. 대소문자 확인
4. GitHub Pages 배포 경로 확인
5. 브라우저 캐시 가능성 확인

## 22. 에이전트 구동용 코어 프롬프트

외부 AI 에이전트, Codex, Claude Code, Cursor Agent, Gemini CLI 등에 작업을 위임할 때 아래 프롬프트를 컨텍스트 최상단에 주입한다.

```text
프로젝트 루트의 AGENTS.md를 최우선 마스터 지침으로 인식하고 모든 행동의 기준으로 삼아라.

이 프로젝트의 최우선 목표는 새 기능 추가가 아니라 기존 홈페이지를 깨뜨리지 않는 것이다. 안정성이 기능보다 우선하며, 수정 범위는 최소화해야 한다.

작업 개시 전 AGENTS.md 전체를 읽고, git status --short를 수행하여 로컬 상태를 점검하라. 실제 파일 구조와 AGENTS.md의 프로젝트 구조가 다르면 작업을 시작하지 말고 차이점을 먼저 보고하라.

다중 파일 수정, 공통 CSS/JS 수정, 배포 작업이 수반될 경우 반드시 프로젝트 상위 폴더에 obaksa-home_backup_YYYYMMDD_HHMMSS 형식으로 원본 전체 복사 백업을 선행하라.

요청 범위 내의 일반 수정, 오타 정정, 메타데이터 보강, 마케팅 문구 정제, 링크 수정, 이미지 경로 수정은 중간 컨펌 없이 진행할 수 있다.

단 git reset --hard, git clean -fd, git push -f, git pull --rebase, .git 폴더 삭제, 프로젝트 폴더 삭제, images/cases 또는 images/gallery 대량 삭제, 외부 설정 파일 전체 교체 같은 파괴적 작업은 절대 임의로 진행하지 말고 사용자 승인을 먼저 획득하라.

CSS 수정 시 전역 선택자 사용을 최소화하고, 신규 컴포넌트에는 ob- 접두사를 포함한 독립 네임스페이스를 사용하라. 월드클락, 플로팅 버튼, FAQ, 히어로, 시공사례 카드가 서로 충돌하지 않도록 스코프를 분리하라.

모든 수정 완료 후 UTF-8 인코딩, 이미지 물리 경로, 하이퍼링크 대상, 모바일 390px/430px/768px 반응형, article data-* 필수 속성 7개, JSON-LD @context 값, sitemap.xml 반영 여부를 검증하라.

검증이 완료되면 git diff --name-only와 git diff --stat으로 변경 범위를 확인하고, 안전한 커밋 메시지로 git add, commit, push를 순차 수행하라. Push 실패나 충돌 발생 시 무리하게 해결하지 말고 즉시 보고하라.

모든 루틴이 종료되면 AGENTS.md의 작업 완료 보고 표준 양식에 맞추어 결과를 사용자에게 명확하게 보고하라.
```
